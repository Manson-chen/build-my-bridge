## 背景与现状

BuildMyBridge 是一个飞书机器人与 Dify（私有部署）集成的中间件项目。

**技术架构：**
- 后端：Spring Boot 单体应用
- 前端：Vue3 SPA
- 数据存储：MySQL + Redis
- 认证方式：飞书 OAuth 扫码登录
- 绑定关系：多对多（一个飞书机器人 ↔ 多个 Dify 应用）

**约束条件：**
- 暂不需要 Spring Cloud 微服务架构
- 暂不支持飞书卡片消息交互
- 暂不支持多租户隔离

## 目标 / 非目标

**目标：**
- 实现飞书机器人与 Dify Chatflow/Workflow 的双向打通
- 提供 Web 管理界面用于配置绑定关系
- 支持多对多绑定和优先级路由
- 记录消息日志供查看和分析
- 集成飞书 OAuth 实现管理员登录

**非目标：**
- 暂不支持微服务架构（单体 Spring Boot 足够）
- 暂不支持飞书卡片消息交互（仅支持文本消息）
- 暂不支持 Dify 知识库检索
- 暂不支持多租户隔离

## 关键设计决策

### 1. 技术栈选择

**决策：** 后端使用 Spring Boot，前端使用 Vue3

**理由：**
- Spring Boot 是 Java 生态最成熟的 Web 框架
- Vue3 国内生态成熟，与飞书/Dify 相关教程丰富
- 用户明确指定

### 2. 数据库设计

**决策：** 使用 MySQL 存储业务数据，Redis 做缓存

**理由：**
- MySQL 成熟稳定，适合结构化数据存储
- Redis 用于缓存飞书调用凭证和路由结果，降低 API 调用频率

**表结构设计：**
- user：管理员账户（通过 OAuth 创建）
- feishu_bot：飞书机器人配置
- dify_app：Dify 应用配置
- binding：多对多绑定关系（带优先级）
- message_log：消息日志

### 3. 消息路由策略

**决策：** 使用优先级路由，默认取最高优先级的绑定

**理由：**
- 实现简单，易于理解
- 满足多对多绑定场景
- 可扩展支持关键字路由（后续版本）

### 4. 飞书消息处理

**决策：** 异步处理消息，支持签名验证和加密消息

**理由：**
- 飞书 Webhook 有超时限制，异步处理更可靠
- 签名验证确保消息来源合法
- 加密 key 可选配置，兼容两种模式

### 5. Dify API 调用

**决策：** 支持 Chatflow 和 Workflow 两种调用方式

**理由：**
- Dify 私有部署支持两种应用类型
- 使用 response_mode=blocking 简化实现

## 风险与权衡

- **风险：飞书 Webhook 超时**
  - 应对措施：使用 @Async 异步处理消息，收到请求立即返回 200

- **风险：Dify 响应时间过长**
  - 应对措施：设置超时时间，超时返回友好错误提示

- **风险：多对多绑定时消息路由不明确**
  - 应对措施：使用优先级字段，数字越大越优先，相同优先级按创建时间排序

- **风险：飞书 token 过期**
  - 应对措施：Redis 缓存 token 并设置 2 小时 TTL，失效后自动刷新
